🔑 OLYMPUS Agent Access Setup

Phase 4 services · what each agent needs to interact · March 2026
Pattern for all services: Every API key, token, and credential gets stored as a Kubernetes Secret in the relevant namespace, then referenced in the LiteLLM config and each agent's environment. Never hardcode credentials in manifests. The agents call services via their internal cluster DNS (http://service.namespace.svc.cluster.local) or via Traefik subdomains when needed.
Email routing: All agent emails at name@ramoneees.com route via Cloudflare Email Routing to the owner's inbox. Each agent has a dedicated address (hermes@, hephaestus@, prometheus@, athena@, plutus@, themis@, mnemosyne@). Catch-all is enabled as a safety net for any unmatched addresses.

Vikunja — Task Backbone
tasks.ramoneees.com
Hermes · creates + delegates tasks Plutus · logs financial tasks Hephaestus · updates task status Prometheus · infra tasks Themis · reads audit trail Mnemosyne · episodic memory
1 Create a dedicated OLYMPUS service account

In Vikunja → Settings → API Tokens → Create Token. Name it olympus-agent. Set permissions: Tasks: Read/Write, Projects: Read/Write, Labels: Read/Write. No expiry or set long expiry.

2 Create one project per agent

Create projects: Hermes, Hephaestus, Prometheus, Athena, Plutus, Themis, OLYMPUS-Audit. Note each project ID (visible in URL). Agents will create tasks scoped to their own project.

3 Create task labels

Create labels: delegated, in-progress, pending-review, approved, rejected, financial, infra, code. Agents use these for workflow state.

Kubernetes Secret — namespace: apps
VIKUNJA_API_URL=https://tasks.ramoneees.com/api/v1
VIKUNJA_API_TOKEN=<token from step 1>
VIKUNJA_PROJECT_HERMES=<project id>
VIKUNJA_PROJECT_PLUTUS=<project id>
VIKUNJA_PROJECT_HEPHAESTUS=<project id>
VIKUNJA_PROJECT_PROMETHEUS=<project id>

Firefly III — Finance
firefly.ramoneees.com
Plutus · primary — read/write transactions Hermes · receives financial alerts Themis · read-only audit
1 Create a Personal Access Token for Plutus

Firefly III → Profile → OAuth → Personal Access Tokens → Create New. Name: plutus-agent. This gives full API access — Plutus needs read + write for transactions, budgets, categories, and accounts.

⚠️ This token must be stored ONLY in the local-only Kubernetes Secret. Never in LiteLLM config or any cloud-accessible store. Plutus is strictly local.

2 Create a read-only token for Themis

Same process, name: themis-readonly. Themis only needs GET access for audit and reporting.

3 Configure webhooks for Hermes

Firefly III → Webhooks → Create. Trigger: After transaction creation. URL: http://n8n.apps.svc.cluster.local:5678/webhook/firefly-transaction. This lets Hermes get notified of new transactions automatically via n8n.

4 Note your account IDs and budget IDs

Plutus needs to know your account IDs to create transactions correctly. Run: GET /api/v1/accounts and note the IDs for your main accounts. Add them to the Plutus system prompt context.

Kubernetes Secret — namespace: apps · LOCAL NODE ONLY (olympus)
FIREFLY_API_URL=https://firefly.ramoneees.com/api/v1
FIREFLY_PLUTUS_TOKEN=<personal access token>
FIREFLY_THEMIS_TOKEN=<readonly token>

Invoice Ninja — Invoicing
invoice.ramoneees.com
Plutus · primary — invoices + clients + payments Hermes · invoice status notifications
1 Generate API token

Invoice Ninja → Settings → API Tokens → Add Token. Name: plutus-agent. Copy the token — it's only shown once.

2 Note your company token

Also needed for API calls: Settings → API Tokens → your company X-API-TOKEN header value. Invoice Ninja requires both the API token and company token in headers.

3 Set up webhook for payment events

Settings → Webhooks → Create. Event: Invoice Paid. URL: http://n8n.apps.svc.cluster.local:5678/webhook/invoice-paid. Hermes gets notified automatically.

Kubernetes Secret — namespace: apps · LOCAL ONLY
INVOICE_NINJA_URL=https://invoice.ramoneees.com/api/v1
INVOICE_NINJA_API_TOKEN=<api token>
INVOICE_NINJA_COMPANY_TOKEN=<company token>

Mattermost — Communications
chat.ramoneees.com
Hermes · posts responses + notifications Plutus · financial alerts only Prometheus · infra alerts Hephaestus · PR notifications
1 Create a bot account for Hermes

Mattermost → System Console → Integrations → Bot Accounts → Add Bot Account. Username: hermes. Display name: 🪽 Hermes. Role: Member. Copy the bot access token.

2 Create dedicated channels

Create channels: olympus-general, olympus-alerts, olympus-finance (private), olympus-infra, olympus-code, olympus-log. Note the channel IDs (visible in channel URL).

⚠️ Make olympus-finance private — Plutus financial alerts should not be visible to all team members.

3 Create incoming webhooks per channel

Integrations → Incoming Webhooks → Add. Create one per channel. These give each agent a direct POST URL to send messages without bot auth complexity for one-way notifications.

4 Enable slash commands or outgoing webhook for you → Hermes

Integrations → Outgoing Webhooks → Add. Channel: olympus-general. Trigger words: @hermes. Callback URL: http://openclaw.olympus.svc.cluster.local:18789/webhook/mattermost. This routes your messages to OpenClaw/Hermes.

MATTERMOST_URL=https://chat.ramoneees.com
MATTERMOST_BOT_TOKEN=<hermes bot token>
MATTERMOST_CHANNEL_GENERAL=<channel id>
MATTERMOST_CHANNEL_ALERTS=<channel id>
MATTERMOST_CHANNEL_FINANCE=<channel id>
MATTERMOST_CHANNEL_INFRA=<channel id>
MATTERMOST_WEBHOOK_GENERAL=<incoming webhook url>
MATTERMOST_WEBHOOK_ALERTS=<incoming webhook url>

Authentik — SSO
auth.ramoneees.com
Hermes · user/session management Prometheus · monitors auth events
1 Create a service account token

Authentik → Admin → Directory → Service Accounts → Create. Name: olympus-agent. Generate a token. This allows Prometheus to read audit logs and Hermes to manage sessions if needed.

2 Connect all Phase 4 services to Authentik SSO

For each running service, create an OAuth2/OIDC provider in Authentik and configure the service to use it. Priority: Vikunja → Gitea → Mattermost → Uptime Kuma. This gives you one login for everything.

3 Set up event webhook for Prometheus monitoring

Authentik → System → Event Rules → Create. Send login failures and suspicious events to n8n → Prometheus → Mattermost #olympus-alerts.

Kubernetes Secret — namespace: infrastructure
AUTHENTIK_URL=https://auth.ramoneees.com
AUTHENTIK_TOKEN=<service account token>

Uptime Kuma — Monitoring
status.ramoneees.com
Prometheus · reads monitor status Hermes · receives downtime alerts
1 Enable API access

Uptime Kuma → Settings → API Keys → Add API Key. Name: prometheus-agent. Prometheus can query monitor status, response times, and incident history via the REST API.

2 Set up Mattermost notification channel

Settings → Notifications → Add → Mattermost. Use the olympus-alerts incoming webhook URL. Enable for all monitors. Hermes receives alerts in #olympus-alerts and can take action.

3 Add monitors for all running services

Add HTTP monitors for: Vikunja, Firefly III, Invoice Ninja, Mattermost, Authentik, Vaultwarden, Homebox, Cloudbeaver, AdGuard, Traefik, Ollama, LiteLLM, OpenClaw. Set check interval to 60s.

Kubernetes Secret — namespace: monitoring
UPTIME_KUMA_URL=https://status.ramoneees.com
UPTIME_KUMA_API_KEY=<api key>

Vaultwarden — Passwords
vault.ramoneees.com
Hermes · secret retrieval only
1 Do NOT give agents direct Vaultwarden API access

Vaultwarden stores all your personal passwords. Agents should never have direct read access to the vault. Instead, store agent credentials in Kubernetes Secrets (which they already have access to) and store your personal passwords in Vaultwarden for human use only.

The correct pattern: secrets agents need → Kubernetes Secrets. Secrets you need as a human → Vaultwarden. These are separate stores for a reason.

2 Optional: Vaultwarden admin API for Prometheus

If you want Prometheus to monitor Vaultwarden health, enable the admin token in Vaultwarden config (ADMIN_TOKEN env var) and give Prometheus read-only access to /admin endpoint for health checks only.

How to create the Kubernetes Secrets
Template — create secrets from CLI (never commit these to git)
# Vikunja
kubectl create secret generic vikunja-agent-secrets \
  --namespace apps \
  --from-literal=VIKUNJA_API_TOKEN=xxx \
  --from-literal=VIKUNJA_API_URL=https://tasks.ramoneees.com/api/v1

# Firefly (local-only — pin to olympus node)
kubectl create secret generic firefly-agent-secrets \
  --namespace apps \
  --from-literal=FIREFLY_PLUTUS_TOKEN=xxx \
  --from-literal=FIREFLY_THEMIS_TOKEN=xxx \
  --from-literal=FIREFLY_API_URL=https://firefly.ramoneees.com/api/v1

# Mattermost
kubectl create secret generic mattermost-agent-secrets \
  --namespace apps \
  --from-literal=MATTERMOST_BOT_TOKEN=xxx \
  --from-literal=MATTERMOST_URL=https://chat.ramoneees.com

# Then reference in your agent Deployment/StatefulSet:
# envFrom:
# - secretRef:
#     name: vikunja-agent-secrets

For GitOps: use Sealed Secrets or External Secrets Operator to encrypt secrets before committing to Gitea. Never commit plain Kubernetes Secrets.
